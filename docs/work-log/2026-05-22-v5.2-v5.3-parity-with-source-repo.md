## 残差: `lib.es5.d.ts` の `ts-type-utils` overlap

the source repo は `the private type-utils lib` が `PropertyKey` / `Partial` / `Required` / `Readonly` / `Pick` / `Record` / `Exclude` / `Extract` / `Omit` / `ReadonlyArray.length` / `ReadonlyArray.[n]` を global 再定義しているため、それらを strict lib 側ではコメントアウトしている (`// This is already defined in ts-type-utils.`)。

当 repo は `ts-type-forge` を採用しており、これらの built-in 型は再定義しない。したがって strict lib 側でこれらを保持する必要があり、コメントアウトすると `noLib: true` + 自前 lib のみという `output/tsconfig.json` の型チェックが破綻する。これは **architectural な差異** であり、ts-type-forge を使う限り当 repo 側で `ts-type-utils` overlap 系のコメントアウトを再現する選択肢はない。

裏返せば、利用者が `the private type-utils lib` 相当のグローバル定義を持ち込んだ場合に duplicate identifier エラーになる可能性はあるが、`ts-type-forge` を peerDep にしている当 repo の設計ではこちらが正である。

## コメント整形差分について

the source repo 側は `prettier-plugin-jsdoc` を併用しており、JSDoc が単行 `/** 短い説明 */` に折りたたまれている。当 repo は同 plugin を使っていないため、複数行 `/**\n   * 短い説明\n   */` のままになる。`prettier-plugin-jsdoc` の採用は別途検討要 (副作用が大きい場合があるため即時導入はしない)。

## 関連変更

- `common.mts`: `typedArrayThisCaptureRegexSource(typeName)` を新設 (`(this|TypeArray)`)。
- `lib.es2023.array.mts`, `lib.es5-typed-array.mts`: 既存の `typedArrayThisRegexSource` 利用を `typedArrayThisCaptureRegexSource` + `$1` 出力に置換。
- `lib.es2017.sharedmemory.mts`: `new (byteLength\??: ...)` / `slice\(begin\??: ...)` を capture group 化。
- `lib.es2015.iterable.mts`: `next(...[value]:)` の変換を TS 5.6+ のみに限定。
- `.cspell.config.yaml`: `strictify` を辞書に追加。
